# 教学玩法重构策划 v0.1

> 项目：StudyGame  
> 文档角色：产品上位重构稿  
> 日期：2026-06-01  
> 作用：统一 `记忆宫殿 + 半开放 RPG + 教材递进` 三条主线，作为当前 PRD、故事线、开发拆解和任务配置维护的上位依据

## 1. 重构目标

这次重构不再把 StudyGame 定义成“带地图和任务的校园英语学习应用”，而是定义成一款儿童生活冒险游戏，英语学习被嵌入探索、照顾、选择、购买、收集和剧情推进中。

一句话定位：

`StudyGame` 是一款以 `home -> school -> town -> transport -> world` 为探索路径、以 `记忆宫殿` 为底层学习结构、以 `半开放养成 RPG` 为外层体验的儿童英语生活冒险游戏。

本次重构要同时解决五个问题：

1. 孩子不能一眼把它识别成“学校 App”。
2. 地图、A-Z 编码和剧情推进必须属于同一个系统，而不是三套并行设计。
3. 英语内容必须从更早基础开始补齐，再自然过渡到现有教材主线。
4. 宠物、换装、商店、奖励和家长确认机制必须成为持续动机，而不是附属装饰。
5. 现有世界地图、A-Z 锚点、家长摘要和任务系统要保留可继承资产，而不是推倒重来。

## 2. 产品新定位

### 2.1 外层体验

孩子感知到的产品不是课时、单词表和题目，而是一个可探索、可照顾、可装扮、可收集、可推进剧情的小镇世界。

前台关键词固定为：

- home
- school
- town
- pet
- shop
- dress up
- help
- trip
- show
- wish

不前台直出的内容：

- 课时
- Unit
- L1 / L2 / L3
- 单词表
- 句型表
- 复习题
- 考试式挑战

这些只保留在家长层、内容配置层、课程审核文档和后续教师/家长解释层。

### 2.2 为什么不是“学校 App”

孩子进入产品后的第一动机必须是：

1. 我想去哪里看看。
2. 我想照顾宠物。
3. 我想帮朋友做点事。
4. 我想买东西或换装。
5. 我想知道小镇下一步会发生什么。

而不是：

1. 我要完成一节英语课。
2. 我要背一组单词。
3. 我要做一道练习。

因此，英语的角色被重新定义为：

`解决生活问题和推动剧情的工具`

而不是：

`前台直接暴露的学习目标`

### 2.3 世界结构

产品的空间主线固定为：

`home -> school -> town -> transport -> world`

含义如下：

- `home`：情绪安全区、起步区、宠物与家园区
- `school`：熟悉区、同伴区、教材高频词承接区
- `town`：生活区、消费区、委托区、社交区
- `transport`：路线区、扩展区、城市边缘认知区
- `world`：文化延展区、旅行分享区、高年级视野扩展区

学校是关键区域，但不再是整个产品的脸。

## 3. 三条主线统一框架

### 3.1 记忆宫殿线

记忆宫殿不是彩蛋系统，而是整套产品的底层学习骨架。

硬规则固定如下：

1. `一字母一主编码`
2. `一主编码一固定位置`
3. `A-Z 主编码表冻结`
4. `新增场景只能扩展场景词和任务词，不得替换主锚点`
5. `home -> school -> town -> transport -> world` 是长期固定路线

这意味着：

- A 仍然是当前已经锁定的 `Apple`
- A-Z 现有主编码表继续有效
- 世界地图中已经绑定的主锚点位置不因玩法扩张而改写
- `music room`、`art room`、`pet shop`、`home decor` 之类新增场景只能增加新的场景词和任务，不得改写主字母编码

#### 3.1.1 A-Z 在产品中的四层功能

1. `字母识别入口`：帮助前期玩家建立字母稳定图像
2. `单词图像入口`：让单词不是抽象文本，而是有地点和图像绑定
3. `地点回忆入口`：从地点反推字母，从字母反推地点
4. `任务与复习回调入口`：后续剧情、商店、宠物、旅行任务不断借旧锚点回调旧知识

#### 3.1.2 开放方式

采用已经锁定的方式：

- 序章完成后，全量 A-Z 锚点正式开放

进一步约束：

- 在序章内，只暴露序章教学所需的强锚点或基础锚点
- 序章完成后，世界地图中的全部 A-Z 锚点都进入正式交互状态
- 当前实现边界：`正式交互状态` 先表示锚点可点击、可进入首访记忆对话；`Memory Spark` 回访提取仍按 pilot 锚点逐步扩展，不等于首版 26 个锚点全部拥有同等回访卡。
- 即使某个锚点尚未进入当前剧情重点，它的位置也从第一天起固定，不因开放进度而搬家

这保证：

- 宫殿稳定
- 回忆稳定
- 后续章节不会反向拆毁前期记忆

### 3.2 半开放 RPG 线

外层产品体验固定为“半开放养成 RPG”，不是完全自由沙盒，也不是强线性课堂流程。

#### 3.2.1 为什么是半开放

完全开放对低龄用户的成本太高，会导致：

- 不知道下一步做什么
- 在地图里乱走但无法理解目标
- 学习内容失去节奏

完全线性又会导致：

- 重新变成课堂壳
- 缺乏探索动机
- 宠物、商店、换装、家园只能变成附属 UI

因此本产品采用：

`一条主线 + 少量支线 + 稳定回访空间`

#### 3.2.2 玩家核心体验

玩家感知到的是：

- 探索
- 委托
- 宠物
- 换装
- 商店
- 收集
- 剧情

学习内容只在这些行为中自然出现。

#### 3.2.3 核心循环

核心循环固定为：

1. 在 `home` 或小镇接到目标
2. 探索地图、对话、观察、购物或照顾宠物
3. 用英语理解提示并完成交互
4. 获得 `Coins / 道具 / 外观 / 宠物材料`；词汇/表达记录只进入家长层和报告层
5. 回家养成、换装、布置、解锁后续剧情

长期循环固定包含：

1. 剧情推进
2. 宠物成长
3. 皮肤/装扮收集
4. 家园和地图解锁
5. 教材难度递进
6. 旧知识点回访

#### 3.2.4 玩法包装原则

任何学习交互都要被包装成生活事件，而不是课程项。

示例：

- 不叫 `校园导览任务`，而叫“陪 Mina 带新朋友认识学校”
- 不叫 `复习挑战`，而叫“节日前的准备活动”或“旅行前的打包检查”
- 不叫 `单词点击题`，而叫“帮店主找对物品”“帮宠物拿到需要的东西”

### 3.3 教材递进线

教材递进采用已经锁定的结构：

- `序章 + 七篇`

#### 3.3.1 序章的角色

序章不是“第 0 篇教材”，而是一个独立的基础层。

序章目标：

1. 补齐三年级下册之前的基础内容
2. 建立 A-Z 和世界地图的最初记忆
3. 建立 home、pet、basic things 的低压力互动体验
4. 让孩子在进入后续教材主线时产生“见过、用过、熟悉”的感觉

序章内容范围固定包含：

- A-Z
- 字母名
- 基础发音感知
- 最核心生活名词
- 最基础动作词
- 最基础指示表达

示例主题：

- home
- pet
- apple
- book
- bag
- bed
- feed
- play
- go
- come
- this / that

#### 3.3.2 七篇的角色

现有七篇主线继续保留，但在本重构中被重新定义为：

`教材内容的剧情化载体`

而不是：

`直接可见的课时序列`

七篇不在本文中重写到关卡级，但其框架继续有效：

1. 小镇新朋友
2. 校园发现日
3. 节日校园周
4. 城市委托簿
5. 计划家与小记者
6. 世界小小行动队
7. 毕业旅行星图

## 4. 玩家核心循环

### 4.1 单次循环

单次游戏循环固定为：

1. 进入 home 或当前地图区域
2. 接到一个游戏化目标
3. 读懂或观察线索
4. 移动、点击、拖拽、购买、选择或对话
5. 完成事件
6. 获得奖励
7. 把奖励投回宠物、外观、家园或剧情推进

这个循环的关键不是“答对题”，而是“让世界发生变化”。

### 4.2 长线循环

长线循环固定为：

1. `剧情` 提供“为什么继续”
2. `宠物` 提供“为什么每天回来”
3. `装扮` 提供“为什么想攒奖励”
4. `商店` 提供“为什么想继续获取 Coins”
5. `家长奖励` 提供“为什么现实世界愿意共同参与”
6. `记忆宫殿` 提供“为什么旧知识会越用越熟”

### 4.3 回访逻辑

每次回访旧区域时，都要尽量做到至少一种：

1. 旧字母锚点再次出现
2. 旧单词在新任务中复现
3. 旧地点承担新的剧情功能
4. 宠物或商店系统把旧词重新调出

这样产品不是线性消耗内容，而是不断在既有宫殿上加深痕迹。

## 5. 教学底层结构

教学底层不再直接以前台课时存在，而采用四层结构。

### 5.1 字母层

内容：

- A-Z
- 字母名
- 基础音感
- 主编码图像

载体：

- 世界地图强绑定锚点
- 序章中的 home / pet / school 周边互动

### 5.2 单词层

内容：

- 生活高频名词
- 地点词
- 宠物词
- 基础物品词
- 动作词

载体：

- 地图探索
- 购物
- 养宠
- 委托

### 5.3 句型层

内容：

- This is...
- That is...
- Where is...?
- I want...
- Can you...?
- I can...
- Go / find / feed / buy / help 相关基础表达

载体：

- NPC 对话
- 选择
- 路线与物品交互

### 5.4 任务层

内容：

- 生活事件
- 城镇委托
- 旅行准备
- 节日活动
- 商店购买
- 宠物照顾

载体：

- 前台全部游戏行为

关系固定为：

`孩子看到的是生活事件、角色请求和冒险目标；任务层仅为内部结构`

`系统运行的是字母层 -> 单词层 -> 表达层 -> 内部任务配置层`

## 6. 世界与章节推进

章节结构固定如下：

1. `序章：Letters, Home, My First Pet`
2. `第一篇：小镇新朋友`
3. `第二篇：校园发现日`
4. `第三篇：节日校园周`
5. `第四篇：城市委托簿`
6. `第五篇：计划家与小记者`
7. `第六篇：世界小小行动队`
8. `第七篇：毕业旅行星图`

每一章在玩法重构稿中都必须用统一四元组来理解。

### 6.1 序章：Letters, Home, My First Pet

- `剧情目标`：让新搬来的孩子先熟悉 home、宠物和最基础世界锚点
- `语言目标`：A-Z、基础名词、基础动作词、最简单指示表达
- `地图主场景`：home、school 首屏周边、少量序章锚点
- `核心玩法形态`：养宠、找物、指认、基础路线探索

### 6.2 第一篇：小镇新朋友

- `剧情目标`：认识小镇伙伴并进入第一圈生活区域
- `语言目标`：问候、时间、天气、基础生活表达
- `地图主场景`：school、garden、部分 town 节点
- `核心玩法形态`：认识伙伴、参加小活动、第一次小镇协作

### 6.3 第二篇：校园发现日

- `剧情目标`：把校园从熟悉地点变成真实可反复使用的生活空间
- `语言目标`：学校词、作息、食物、动物、能力、规则
- `地图主场景`：school 核心簇、home 回访点
- `核心玩法形态`：导览、整理、照顾、规则理解

### 6.4 第三篇：节日校园周

- `剧情目标`：让校园从静态地点升级为“正在发生事情的节日现场”
- `语言目标`：位置、课程、星期、活动和进行时
- `地图主场景`：school、park、节日节点
- `核心玩法形态`：节日准备、活动观察、现场判断

### 6.5 第四篇：城市委托簿

- `剧情目标`：玩家开始独立处理 town 里的生活委托
- `语言目标`：购物、点餐、借物、宠物、情绪、节日生活
- `地图主场景`：town 商店群、pet shop、restaurant、bookshop
- `核心玩法形态`：购买、选择、照顾、帮助他人

### 6.6 第五篇：计划家与小记者

- `剧情目标`：从“帮忙的人”升级成“能计划和记录的人”
- `语言目标`：计划、研究、问路、月份、过去时
- `地图主场景`：town、transport、information 节点
- `核心玩法形态`：路线、制作、采访、调查

### 6.7 第六篇：世界小小行动队

- `剧情目标`：把视野从个人生活扩展到更大的公共行动
- `语言目标`：将来时、建议、比较、健康、环保、节日
- `地图主场景`：transport、world 边框延展区、公共活动节点
- `核心玩法形态`：准备、协作、展示、计划实施

### 6.8 第七篇：毕业旅行星图

- `剧情目标`：把小学阶段英语能力整合进一次毕业旅行分享
- `语言目标`：综合表达、比较、建议、旅行与世界介绍
- `地图主场景`：world 边框区、全图回访
- `核心玩法形态`：整合展示、旅行准备、最终回访

## 7. 系统设计

### 7.1 任务与前台表达

前台统一采用偏游戏化命名，不把学习概念摆到孩子面前。

允许的前台表达：

- Quest
- Wish
- Help
- Trip
- Show
- Shop
- Pet Care

不允许直接替代孩子前台界面的表达：

- Lesson
- Unit
- Vocabulary
- Pattern
- Review
- Test

这意味着：

- `QuestDiary` 应重构表达为偏 `Quest Diary`
- 系统内部仍可保留任务配置化结构，但显示层必须转义

### 7.2 宠物系统

宠物系统首版定义为轻养成，不做战斗化。

固定状态：

- 饥饿
- 清洁
- 心情
- 亲密度

固定互动：

- 喂食
- 清洁
- 陪玩
- 睡觉

产品职责：

1. 作为长期回访动机
2. 作为低压学习触发器
3. 作为序章和前期章节的重要场景载体

宠物相关词汇和动作词优先进入：

- 序章
- 第一篇
- 第四篇部分生活委托

### 7.3 装扮与外观系统

装扮系统是长期目标，不是一次性奖杯。

固定承载内容：

- 角色服装
- 配饰
- 包
- 房间角落装饰
- 宠物外观或用品

作用固定为：

1. 提供持续目标
2. 稀释做题感
3. 让奖励与表达自我绑定

### 7.4 商店系统

商店固定拆成三类：

1. `Pet Shop`
2. `Clothes Shop`
3. `Home Decor / General Store`

商店作用固定为：

1. 消耗 `Coins`
2. 购买宠物耗材
3. 购买外观与装饰
4. 在真实消费场景中复现英语词汇

商店不是单独 UI 功能，而是世界地图和剧情的一部分。

### 7.5 经济系统

经济结构固定为双层货币：

1. `Coins`
2. `Parent Bonus`

#### 7.5.1 Coins

来源：

- 完成事件
- 探索
- 照顾宠物
- 剧情推进
- 常规活动

用途：

- 宠物耗材
- 普通装扮
- 普通装饰

#### 7.5.2 Parent Bonus

来源：

- 家长确认后追加

用途：

- 稀有奖励
- 限定外观
- 阶段性激励

#### 7.5.3 经济边界

不允许把整套系统做成：

- 刷题换金币
- 单一货币覆盖一切
- 家长无限补给常规成长

### 7.6 家长层

家长层和孩子层职责必须拆开。

孩子层看到：

- 任务
- 地图
- 宠物
- 商店
- 奖励

家长层看到：

- 学过哪些词和句型
- 当前剧情进度
- 哪些知识点需要回访
- 是否发放 `Parent Bonus`

家长层的价值是：

1. 解释学习成果
2. 连接现实激励
3. 保持信任

## 8. 家长层与奖励层

### 8.1 家长层的表达原则

家长看到的是：

- 孩子今天做了什么
- 用到了哪些词和表达
- 哪些知识点还值得回访
- 是否达到阶段性激励条件

而不是只看到：

- 玩了多久
- 点了多少次
- 过了几关

### 8.2 奖励层的双目标

奖励层同时服务两个对象：

1. 孩子：提供持续玩下去的内在动机
2. 家长：提供可确认、可解释、可控制的外部激励接口

### 8.3 为什么保留家长奖励确认

家长奖励不只是给币，而是把现实世界和游戏行为连接起来。

它的作用是：

1. 把学习成果转译成可见激励
2. 让家长参与但不过度占领游戏
3. 避免孩子把所有成长都理解成“做题刷数值”

## 9. 对现有项目的影响

### 9.1 保留的方向

以下现有方向应明确保留：

- 世界地图主结构
- `home -> school -> town -> transport -> world` 方向
- 对话系统
- 点击/拖拽/轻交互系统
- 家长摘要
- A-Z 锚点和世界地图热点体系

### 9.2 旧口径与当前表达

以下表达不再作为儿童前台产品语言，只能出现在内部兼容、报告 ID、迁移说明或文档审计语境中：

- `QuestDiary`：已重构为 `Quest Diary` 冒险日志，前台不显示内部节点名。
- `School Tour`：旧教学口径，前台统一改成 `Walk With Mina` 或同伴陪伴事件。
- `ReviewChallenge`：旧内部/报告兼容口径，前台统一包装成 `Story Show`、展演、旅行准备或宠物/城镇事件，不再保持考试感入口。
- 现有 MVP 旧关卡/任务资产：当前叙述统一重述为“生活事件/委托”，而非单纯课堂任务

### 9.3 对现有 MVP 的解释方式

现有 MVP 不是废弃，而是需要被重新命名和重新包装。

例如：

- 校园导览，不再被表述为课堂目标，而是迎新或陪伴事件
- 整理教室，不再被表述为语法练习，而是帮助朋友整理空间
- 花园找鸟，不再被表述为识图题，而是探索和观察事件
